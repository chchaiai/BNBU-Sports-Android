# BNBU Sports Android Week2 周报

## 周目标是什么

1. 以《BNBU Sports Design System v1.0》为唯一视觉标准，完成学生 Android 端视觉升级，并支持浅色、深色和跟随系统三种主题模式，默认浅色。
2. 在 Week1 版本基础上补齐 Week2 学生端需求：课程学期与历史课程、每日一次打卡、1h/2h 时长、运动项目矩阵、AI 审核状态、三类免测/免打卡申请、服务端耐力跑换算和入学年份年级推算。
3. 建立 Android 独立后端、数据库和公网端口，完成 LightCOS 图片/视频上传链路。
4. 保持 Android 仅面向学生使用，不实现教师端、管理端和 Web 端管理能力。

## 周完成进度汇报

### 已完成

- Android 视觉系统已按设计文档升级，统一颜色、字体、形状、卡片、按钮和页面布局。
- 已支持浅色、深色、跟随系统三种主题模式，默认浅色，并将用户选择持久化。
- 登录支持学号或邮箱，Android 使用 `clientType=mobile` 调用真实后端。
- 课程页支持当前学期、历史课程折叠和课程状态展示。
- 打卡时长固定为 1h/2h；增加运动项目矩阵和“其他”自定义项目。
- Android 本地提前判断当日是否已经提交；后端同时增加每日一次校验。
- 上传限制已实现：最多 6 张图片、每张不超过 8MB；最多 1 个视频、不超过 100MB。
- AI 审核状态与人工审核状态分离，学生端可展示 AI 审核中、正常、异常和转人工等状态。
- 跑步免测、校队免打卡、社团免打卡三类流程已接入，校队/社团申请支持组织、证明和审核状态。
- 耐力跑成绩由服务器规则计算，Android 只提交数据并展示服务器结果。
- 新增 `admission_year`，年级按自然年在服务器端推算；体测成绩仍以后端为唯一权威来源。
- 网络失败不再伪造成功结果，上传、提交和后端错误会在 Android 端明确展示并支持重试。
- 已生成 Debug APK，并通过 Kotlin 编译和完整 `assembleDebug` 构建。

### 云端部署已完成

## 遇到哪些困难，踩到了哪些坑

- 上周使用的是共享 `96` 端口，本周确认团队没有独立后端人员，因此 Android 开发同时需要负责 Node.js 后端、数据库迁移、COS 和 Nginx 部署。
- 不能直接改共享数据库和 `bnbu-api-v1`，因此先备份生产数据库，再复制到独立数据库并执行迁移，降低对其他端的影响。
- 私有 COS 不能返回服务器本地 `/uploads/...` 路径，必须返回签名 URL，并可同时返回 `cosKey`。
- Nginx 默认请求体限制无法支持 100MB 视频，需要单独设置 `client_max_body_size 120m` 和上传超时。
- Android 过去存在提交失败但界面仍显示成功的风险，本周改为以后端响应为准。
- 宝塔网页终端粘贴较长命令时曾混入历史命令，后续改用短命令、临时文件和脱敏输出完成验证。

## 是否需要和其他端人员合作协调

- 需要负责人确认并安排 Week2 完整人工验收，结果请记录为通过、失败或待确认。
- Android 学生端不实现 Web 管理能力；Web 端继续负责学生上传后备网页和教师/管理员管理功能。
- 后续合并接口时，需要统一 Android 独立后端与团队其他端的 API、数据库迁移和字段定义。
- COS 密钥、数据库密码等敏感配置只保留在服务器 `.env`，不得提交到 GitHub 或在周报中公开。
